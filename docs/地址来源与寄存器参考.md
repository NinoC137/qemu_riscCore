# 地址来源与寄存器参考

## 结论先说

我前面提到的这些地址，不是拍脑袋写的，主要来自 3 类信息源：

1. **QEMU `virt` 板级实现源码**
   - 用来确认 **设备基地址 / 区间大小 / QEMU 实际挂了什么设备**
2. **QEMU 串口与 ACLINT 实现源码**
   - 用来确认 **16550 UART 的寄存器编号语义**、**ACLINT/CLINT 的 timer 偏移**
3. **QEMU 官方 `virt` 文档 / 运行时 DTB**
   - 用来确认 **机器模型语义**，以及提醒：**真正运行时最权威的发现方式是 DTB**

对你当前这个项目，最重要的结论是：

- UART0 基地址：`0x10000000`
- UART0 区间大小：`0x100`
- CLINT 基地址：`0x02000000`
- CLINT 区间大小：`0x10000`
- QEMU `virt` 上 UART 是 **NS16550 compatible UART**
- QEMU `virt` 上 machine timer 常见使用方式可按：
  - `mtimecmp(hart0)` = `0x02004000`
  - `mtime` = `0x0200BFF8`

## 一、我采用的主要信息源

### 1. QEMU 官方文档：`virt` board

来源：
- https://www.qemu.org/docs/master/system/riscv/virt.html

这份文档说明了：

- `virt` 是 QEMU 推荐的 RISC-V 通用虚拟平台
- 它带有：
  - CLINT
  - PLIC
  - **1 个 NS16550 compatible UART**
- QEMU 会自动生成并传递 **DTB** 给 guest
- **guest 应优先通过 DTB 发现设备地址、中断号和配置**

这点很重要：

**文档告诉你“这台机器上有什么”；DTB 告诉你“这次运行时这些东西具体在哪里”。**

如果你后面要从“硬编码 bring-up”走向“更正式的内核”，建议最终以 DTB 为准，而不是永远手写魔数。

---

### 2. QEMU `virt` 板源码：设备基地址

来源：
- `qemu/hw/riscv/virt.c`
- GitHub: https://github.com/qemu/qemu/blob/master/hw/riscv/virt.c

我从源码里抓到的关键条目是：

- `VIRT_CLINT` → `{ 0x2000000, 0x10000 }`
- `VIRT_UART0` → `{ 0x10000000, 0x100 }`
- `VIRT_PLIC` → `{ 0x0c000000, ... }`
- `VIRT_RTC` → `{ 0x00101000, 0x1000 }`

另外，QEMU 初始化 UART 时有一段关键调用：

- `serial_mm_init(system_memory, s->memmap[VIRT_UART0].base, 0, ..., 399193, ..., DEVICE_LITTLE_ENDIAN);`

这句很有价值，因为它说明：

- UART 基地址确实来自 `VIRT_UART0`
- **寄存器 shift = 0**
- 说明 UART 寄存器就是按 **字节偏移 0..7** 访问，而不是每个寄存器再乘 4 或乘 8

所以你代码里这种访问方式：

- `base + 0x00`
- `base + 0x01`
- `base + 0x02`
- ...

在 QEMU `virt` 这个 UART 上是对的。

---

### 3. QEMU 串口实现源码：16550 寄存器语义

来源：
- `qemu/hw/char/serial.c`
- `qemu/include/hw/char/serial.h`
- GitHub:
  - https://github.com/qemu/qemu/blob/master/hw/char/serial.c
  - https://github.com/qemu/qemu/blob/master/include/hw/char/serial.h

这里可以确认两件事。

#### 3.1 它确实是 16550A 兼容模型

`serial.h` 里直接写着：

- `QEMU 16550A UART emulation`

#### 3.2 寄存器偏移就是 0..7

`serial.c` 里处理 MMIO 读写时有：

- `assert(size == 1 && addr < 8);`
- 然后 `switch(addr)` 分别处理 `case 0` 到 `case 7`

这就对应了常见 16550 编程模型：

- `0`：RBR / THR / DLL
- `1`：IER / DLM
- `2`：IIR / FCR
- `3`：LCR
- `4`：MCR
- `5`：LSR
- `6`：MSR
- `7`：SCR

所以我之前写 UART 代码时用的这些偏移，来源就是这里 + 16550 标准寄存器模型。

---

### 4. QEMU ACLINT/CLINT 实现：timer 偏移

来源：
- `qemu/include/hw/intc/riscv_aclint.h`
- `qemu/hw/riscv/virt.c`
- GitHub:
  - https://github.com/qemu/qemu/blob/master/include/hw/intc/riscv_aclint.h
  - https://github.com/qemu/qemu/blob/master/hw/riscv/virt.c

这里能拿到非常关键的默认值：

- `RISCV_ACLINT_SWI_SIZE = 0x4000`
- `RISCV_ACLINT_DEFAULT_MTIMECMP = 0x0`
- `RISCV_ACLINT_DEFAULT_MTIME = 0x7ff8`
- `RISCV_ACLINT_DEFAULT_MTIMER_SIZE = 0x8000`
- `RISCV_ACLINT_DEFAULT_TIMEBASE_FREQ = 10000000`

而 `virt.c` 里又能看到：

- CLINT 基地址是 `0x02000000`
- mtimer 是在 `CLINT base + RISCV_ACLINT_SWI_SIZE` 上创建的

所以对于 hart0，常见计算方式就是：

- `mtimer_base = 0x02000000 + 0x4000 = 0x02004000`
- `mtimecmp = mtimer_base + 0x0 = 0x02004000`
- `mtime = mtimer_base + 0x7ff8 = 0x0200bff8`

这也是很多 bare-metal 例子里常写的：

- `mtimecmp @ 0x02004000`
- `mtime @ 0x0200BFF8`

## 二、给你当前工程最实用的地址总结

### 1. UART0

建议按 QEMU `virt` 里的 NS16550 兼容 UART 处理。

- 基地址：`0x10000000`
- 区间大小：`0x100`
- 寄存器 shift：`0`
- 字节序：little-endian

常用寄存器偏移：

- `0x00`：RBR / THR / DLL
- `0x01`：IER / DLM
- `0x02`：IIR / FCR
- `0x03`：LCR
- `0x04`：MCR
- `0x05`：LSR
- `0x06`：MSR
- `0x07`：SCR

你当前最小输出常用到的是：

- `THR @ base + 0x00`：发送字符
- `LSR @ base + 0x05`：查看发送是否 ready

常用 bit：

- `LSR.THRE = 0x20`：发送保持寄存器空
- `LSR.TEMT = 0x40`：发送器完全空
- `LCR.DLAB = 0x80`：切换 divisor latch 访问

---

### 2. CLINT / Machine Timer

对你现在的最小 M-mode 内核，先按 machine timer 用就行。

- CLINT 基地址：`0x02000000`
- 区间大小：`0x10000`

对 hart0 最常用的地址：

- `mtimecmp(hart0)`：`0x02004000`
- `mtime`：`0x0200BFF8`

这背后的结构是：

- 前面 `0x4000` 左右给 software interrupt / msip 一类区域
- 后面 `0x8000` 的 mtimer 区域里有：
  - 每 hart 的 `mtimecmp`
  - 全局 `mtime`

如果你只做单核 bring-up，可以先只关心 hart0。

---

### 3. PLIC

- 基地址：`0x0C000000`

但你当前阶段先做 **machine timer interrupt**，短期还不需要先碰 PLIC。

PLIC 更像是你后面接外设中断时再进入的阶段。

---

## 四、我建议你以后怎么查地址，避免凭记忆写魔数

### 最推荐顺序

#### 第一优先级：看运行时 DTB

因为 QEMU 官方文档已经明确说了：

- `virt` 会生成 DTB
- guest 应从 DTB 发现地址和中断配置

所以从工程演进角度，**DTB 是运行时最权威来源**。

#### 第二优先级：看 QEMU `virt.c`

因为它定义了板级内存图，是“QEMU 这台虚拟机到底怎么搭的”的源头。

#### 第三优先级：看设备实现源码 / 官方设备手册

例如：

- UART 寄存器偏移：看 16550 模型 / `serial.c`
- timer 偏移：看 ACLINT/CLINT 实现和 RISC-V 平台约定
