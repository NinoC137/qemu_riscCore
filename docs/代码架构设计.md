# qemu-riscv-embedded 整体代码架构设计

## 1. 设计目标

本文件用于定义项目的整体代码组织方式，目标不是直接给出实现，而是先把代码边界、职责划分与模块依赖梳理清楚，避免后续开发过程中出现：

- 平台代码与内核代码耦合过深
- 驱动与设备树描述混杂
- 启动链路职责不清
- 安全域与非安全域代码边界模糊
- 测试、工具、配置散落难以维护

本项目建议采用“**平台层 / 启动层 / 内核层 / 驱动层 / 用户态层 / 安全域层 / 工具与测试层**”的模块化分层结构。

---

## 2. 顶层目录结构

当前工程建议使用如下目录体系：

- `docs/`：项目文档、架构说明、设计草案
- `platform/`：QEMU 平台定义与 SoC/Board 规格
- `boot/`：启动链路相关，包括固件、U-Boot 适配、启动脚本
- `kernel/`：主 OS 内核代码
- `drivers/`：通用驱动与设备接口实现
- `user/`：用户态初始化程序、基础库与测试程序
- `secure/`：安全域、RTOS、域间通信与隔离策略
- `tests/`：测试代码与验证场景
- `tools/`：镜像生成、运行、调试、设备树辅助工具
- `scripts/`：自动化脚本
- `configs/`：各模块配置文件
- `third_party/`：外部依赖与第三方源码镜像
- `build/`：构建输出目录
- `artifacts/`：镜像、dtb、日志等产物归档
- `temp/`：临时文件

---

## 3. 分层原则

## 3.1 platform 层
负责定义“机器是什么”。

这一层回答：
- 四核 RISC-V 平台如何建模
- 板级地址空间如何布局
- 哪些 MMIO 设备挂接到总线
- 设备树如何描述平台资源

这层不应承担：
- OS 内核策略
- 调度逻辑
- 系统调用实现

### 建议子目录
- `platform/qemu/machine/`：machine/board 级定义
- `platform/qemu/devices/`：QEMU 设备模型
- `platform/qemu/bus/`：总线、设备挂接、地址映射辅助逻辑
- `platform/qemu/fwcfg/`：boot 参数、固件配置、镜像传递机制
- `platform/qemu/dt/`：设备树模板、生成与适配说明
- `platform/spec/`：SoC 规格文档、地址映射、中断分配、寄存器草案

### 设计建议
- machine 与 device 必须分层
- 设备树描述与设备模型保持一一对应
- 地址映射、中断号、compatible 字符串必须在 spec 中统一定义

---

## 3.2 boot 层
负责定义“系统如何启动”。

这一层回答：
- reset 后第一段代码在哪里
- 固件如何初始化 CPU 与最小运行环境
- U-Boot 如何装载内核与 DTB
- 如何把控制权交给主 OS 或安全域系统

### 建议子目录
- `boot/firmware/`：早期固件、启动辅助逻辑、平台 bring-up 文档
- `boot/uboot/`：U-Boot 板级适配说明与相关补丁/配置
- `boot/scripts/`：镜像打包、启动命令、启动介质准备脚本

### 设计建议
- boot 层只做启动，不承载长期运行时逻辑
- DTB 的来源与传递流程要在这里固定下来
- 安全域与普通域的启动顺序也要在这里定义接口

---

## 3.3 kernel 层
负责定义“主 OS 如何运行”。

这是项目的核心层，承担：
- trap/异常/中断框架
- 分页式内存管理
- 任务管理与调度
- 系统调用框架
- SMP 支撑
- 内核基础设施

### 建议子目录
- `kernel/arch/riscv/boot/`：RISC-V 架构相关早期启动代码
- `kernel/arch/riscv/trap/`：陷入、异常、中断入口与分发
- `kernel/arch/riscv/mm/`：页表、satp、TLB 等架构相关内存管理
- `kernel/arch/riscv/smp/`：多核启动、IPI、核间协作
- `kernel/arch/riscv/context/`：上下文切换、寄存器保存恢复
- `kernel/arch/riscv/syscall/`：系统调用 ABI 对接
- `kernel/core/init/`：内核初始化流程
- `kernel/core/task/`：任务控制块、进程/线程抽象
- `kernel/core/sched/`：调度器与运行队列
- `kernel/core/timer/`：系统时钟与 tick 管理
- `kernel/core/ipc/`：基础内核 IPC 能力
- `kernel/mm/`：与架构无关的内存管理抽象
- `kernel/fs/`：文件系统或最小 VFS 预留
- `kernel/drivers/`：内核内部使用的驱动适配层
- `kernel/lib/`：内核基础库
- `kernel/include/`：内核公共头文件

### 分层建议
- `arch/riscv/` 中放“必须依赖 RISC-V 架构”的内容
- `core/` 中放“OS 通用机制”
- `mm/` 中放“架构无关策略和抽象”
- `kernel/drivers/` 只保留和内核强绑定的驱动适配，不要与顶层 `drivers/` 重复堆叠职责

---

## 3.4 drivers 层
负责定义“设备如何被软件访问”。

这里建议把驱动做成一个相对独立层，便于后续：
- 从轮询切换到中断驱动
- 从裸接口切换到设备树匹配
- 从最小驱动扩展到更完整模型

### 建议子目录
- `drivers/uart/`
- `drivers/can/`
- `drivers/flash/`
- `drivers/timer/`
- `drivers/rtc/`
- `drivers/interrupt/`
- `drivers/common/`

### 职责建议
- `drivers/common/`：寄存器访问、设备基类、错误码、probe 框架等共性逻辑
- `drivers/interrupt/`：PLIC/CLINT 或其他中断相关适配
- 各设备目录：设备初始化、probe、读写接口、中断处理、设备树匹配

### 重要边界
- platform 层定义“硬件长什么样”
- drivers 层定义“软件怎么访问它”
- kernel 层定义“系统何时、为何使用它”

---

## 3.5 user 层
负责定义“用户空间怎么存在”。

这一层虽然不是最早实现，但需要尽早预留结构，因为它会反向影响：
- 系统调用接口
- 进程模型
- init 启动方式
- 最小 libc 或运行时支持

### 建议子目录
- `user/init/`：init 进程
- `user/libc/`：最小用户态运行库
- `user/apps/`：测试应用、演示程序
- `user/include/`：用户态头文件与 syscall 接口声明

### 设计建议
- init 应独立成模块，不要直接混在测试程序中
- 用户态库与内核 syscall 编号/ABI 要通过公共规范文件绑定
- 后续的 shell、测试 app 都建议放在 `apps/`

---

## 3.6 secure 层
负责定义“安全域系统如何存在”。

这是项目后期重点，但目录应提前预留，避免后续大重构。

### 建议子目录
- `secure/rtos/`：安全域 RTOS 接入与适配层
- `secure/monitor/`：安全监控、切换控制、可信管理逻辑
- `secure/ipc/`：安全域与非安全域通信
- `secure/policy/`：资源隔离策略、访问控制、域划分规则
- `secure/include/`：公共接口头文件

### 设计建议
- 不要一开始就把 secure 层塞进 kernel 主目录里
- 安全域与普通域之间应通过接口契约交互，而不是互相直接调用内部实现
- `policy/` 可以作为后续实验不同隔离方案的容器

---

## 3.7 tests 层
负责定义“你如何证明系统在工作”。

### 建议子目录
- `tests/unit/`：基础模块单元测试
- `tests/integration/`：多模块联调测试
- `tests/boot/`：启动链路测试
- `tests/syscall/`：用户态与 syscall 验证
- `tests/smp/`：多核相关测试
- `tests/security/`：安全域与隔离测试

### 设计建议
- 每完成一个里程碑，就补对应测试，而不是最后统一补
- 测试目录结构最好与里程碑一致，方便回归验证

---

## 3.8 tools / scripts / configs 层
这一层负责降低工程摩擦。

### `tools/`
用于承载辅助程序：
- `tools/image/`：镜像打包与转换工具
- `tools/dtb/`：设备树检查、转换、辅助生成工具
- `tools/debug/`：日志解析、trace 辅助、调试脚本
- `tools/run/`：本地启动器、仿真运行辅助

### `scripts/`
用于放自动化脚本，例如：
- 一键启动开发环境
- 生成测试矩阵
- 打包构建产物
- 日志归档

### `configs/`
建议集中配置，避免散落：
- `configs/qemu/`
- `configs/uboot/`
- `configs/kernel/`
- `configs/rtos/`

这样可以保证“配置”与“代码”解耦，方便版本化管理。

---

## 4. 模块依赖关系建议

建议遵循以下依赖方向：

`platform -> boot -> kernel -> user`

以及横向支撑：

- `drivers` 被 `kernel` 和部分 `boot/secure` 使用
- `secure` 与 `kernel` 平行，不建议深度交叉依赖
- `tools/scripts/configs/tests` 为外围支撑层

更准确地说：

- `platform` 不依赖 `kernel`
- `kernel` 可以读取 `platform/spec` 输出的规范，但不应反向决定平台建模
- `drivers` 依赖平台暴露的硬件规范与设备树协议
- `user` 依赖 `kernel` 的 syscall ABI
- `secure` 依赖平台资源划分规则，但应保持相对独立演化

## 6. 文件组织建议

为了让后续代码更容易维护，建议你尽早统一这些规则：

### 6.1 命名规则
- 目录名尽量按功能命名，不按个人习惯命名
- 文件名优先体现职责，如：
  - `uart_regs.*`
  - `uart_driver.*`
  - `sv39_pagetable.*`
  - `task_scheduler.*`

### 6.2 头文件边界
- 公共头文件进入各模块 `include/`
- 私有实现细节不要轻易暴露到全局 include
- 尽量减少跨层直接 include

### 6.3 接口优先
每个大模块建议先有：
- 对外接口定义
- 数据结构说明
- 生命周期说明
- 错误处理约定

而不是一上来就堆实现文件。

## 8. 结论

这个代码架构的核心思想可以概括为一句话：

**用 platform 定义机器，用 boot 打通启动，用 kernel 实现主 OS，用 drivers 管理设备，用 user 承载用户态，用 secure 扩展可信域，用 tests/tools/configs 保证工程可持续推进。**

这样划分的好处是：
- 模块边界清楚
- 后续扩展外设容易
- 安全域不会把主内核结构搅乱
- 更适合你这种“先做研究原型、再逐步增强”的项目路线
