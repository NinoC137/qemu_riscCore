# qemu-riscv-embedded 开发手册

## 1. 项目目标

本项目建议按“从能启动，到能调度，到能跑用户态”的顺序推进。

推荐主线：

1. 最小启动闭环
2. trap / interrupt
3. 内存管理
4. 任务调度
5. 系统调用
6. 用户态 init
7. 多核
8. 安全域 / RTOS

当前最重要的不是功能多，而是建立一个稳定、可验证、可逐步演进的内核骨架。

---

## 2. 当前阶段判断

你现在已经完成：
- 裸机入口
- BSS 清零
- `kmain` 进入
- 串口输出
- 链接与 QEMU 启动验证

这意味着项目已经跨过“工具链/链接/入口不稳定”的阶段，下一步应进入 **内核基础设施阶段**。

不建议现在立刻做：
- fork/exec
- 多核调度
- 安全域隔离
- 大量外设建模

这些都应建立在 trap、timer、内存管理骨架稳定之后。

---

## 3. 最佳下一步

### 第一优先级：整理最小内核骨架

建议你下一步先完成 4 件事：

### 3.1 平台常量集中管理
把硬编码地址从 `kernel.cpp` 移出。

建议新增：
- `platform/memmap.h`
- `platform/riscv.h`

至少先集中：
- UART 基地址
- CLINT/PLIC 基地址（如果后面要做中断）
- 内核加载地址
- 页大小、栈大小等常量

目标：后续改平台时，不需要到处搜魔数。

### 3.2 UART 驱动最小封装
不要让 `kmain` 直接操作 MMIO。

建议先抽成：
- `drivers/uart/uart.h`
- `drivers/uart/uart.cpp`

最小接口：
- `uart_init()`
- `uart_putc(char)`
- `uart_puts(const char*)`

目标：把“平台寄存器细节”和“内核逻辑”分开。

### 3.3 trap 入口与异常打印
这是最值得优先做的下一步。

至少先建立：
- `mtvec/stvec` 设置
- trap 入口汇编
- C 层 trap handler
- 打印 `mcause/scause`、`mepc/sepc`、`mtval/stval`

目标：以后无论页错误、非法指令、syscall、中断，先能看见。

没有 trap 基础设施，后面很多问题会变成“直接死机，无日志”。

### 3.4 timer interrupt
在 trap 框架建立后，立刻接一个最小 timer 中断。

目标不是马上调度，而是先验证：
- 能定时触发
- 能进 handler
- 能打印 tick
- 能重新装载下一次中断

只要 timer 通了，调度器就有了时钟源。

## 4. 建议的近期开发顺序

### Phase A：稳定 bring-up 基线
- 清理启动代码
- 平台常量集中化
- UART 抽驱动
- 保持构建脚本稳定

交付标准：
- `make clean && make` 稳定成功
- QEMU 启动稳定打印
- 目录边界清楚

### Phase B：trap / interrupt
- trap 向量入口
- 寄存器保存/恢复
- trap C handler
- timer interrupt
- 基本 panic 打印

交付标准：
- 能看见异常原因
- 能看见周期 tick

### Phase C：内存管理雏形
- 页表数据结构
- 内核恒等映射或高地址映射
- 打开分页
- 简单物理页分配器

交付标准：
- 内核在页表下稳定运行
- 能处理最基础页错误

### Phase D：任务与调度
- task/thread 控制块
- 上下文保存恢复
- 基于 timer 的调度切换
- 先单核 round-robin

交付标准：
- 两个内核线程可切换

### Phase E：用户态与 syscall
- S/U 模式边界
- trap 返回用户态
- syscall 分发
- `write`
- `init`

交付标准：
- 用户态程序能打印字符

---

## 5. 当前最推荐你立刻做的一个小目标

如果只选一个，我建议：

**下一步先做 trap + timer。**

原因：
- 它是后续调度、syscall、异常处理的共同基础
- 成本不高，但收益极大
- 一旦能稳定打印 trap 信息，后面调试效率会明显提升

可以把这个小目标拆成：

1. 建 trap 向量入口
2. 故意触发一个异常并打印原因
3. 接入 timer interrupt
4. 每次 timer 中断打印一个 tick 计数

做到这一步，你的项目就从“能启动的 demo”进入“可演进的小内核”。

## 6. 目录演进建议

当前目录可继续保留，但建议逐步形成下面的结构意识：

- `boot/`：只放启动和早期汇编
- `platform/`：地址映射、CSR 常量、板级配置
- `drivers/`：设备驱动
- `kernel/arch/riscv/`：trap、context、页表等架构相关代码
- `kernel/core/`：调度、内存、syscall 等内核通用核心
- `user/`：用户态测试程序
- `docs/`：专题设计文档，不堆大而空的总述

也就是说，后面最好把 `kernel/` 再细分成：
- `arch`
- `core`

这样页表、trap、上下文切换这些“强架构相关内容”就不会和调度器/对象管理混在一起。

---

## 7. 开发约束建议

建议默认遵守：

- 先单核，后 SMP
- 先 M/S 模式内核，后 U 模式
- 先最小机制，后泛化抽象
- C++ 只做组织，不做花哨运行时依赖
- 避免异常、RTTI、隐式重型运行库依赖

一句话：

**先把内核做“硬”，再把工程做“漂亮”。**
